# エラー伝搬に MAYBE マクロを使う

## 日付

- 判断時期: BM時代に部分的に使用開始、EBM移行時（2025年頃）に全域適用
- 文書化: 2026-04-15

## 判断

エラー伝搬に C++ の例外や std::expected のモナディック操作ではなく、
MAYBE(x, expr) 形式のマクロを使う。

## 動機

- Rust の `?` 演算子と同等のことをやりたい。
  式を評価して失敗したらエラーで即 return、成功したら結果を変数に束縛する。
- std::expected のモナディック操作（and_then 等）は C++ だと `and_then([&](auto&& x){ ... })`
  のように記述量が多い。間にロジックを付け足そうとするとコードの大幅な書き換えが必要になる。
  見た目を良くするにはちょっとした処理でも関数に切り出す必要がある。
- マクロなしだと `auto x = expr; if(!x) { return x; } auto& use = *x;` が大量増殖する。
  MAYBE マクロはこれを1行にまとめる。
- あとから「失敗しうるが上位では無視したい」処理を付け加えるのが容易。
  マクロなら呼び出し側のコード構造を変えずに済む。

## 具体例

```cpp
// MAYBE マクロ: 失敗したら即 return
MAYBE(result, some_fallible_operation());
// result はここでは成功値として使える

// これが無いと毎回こう書く必要がある
auto result = some_fallible_operation();
if (!result) { return result; }
auto& result_val = *result;
```

## これは X を意味しない

- 例外が絶対ダメというわけではない。実際 brgen の parser では例外を使っている。
  今考えれば例外でも良かったかもしれないが、現状マクロで動いているので変える理由もない。
- マクロの多用を一般的に推奨しているわけではない。
  C++ に Rust の `?` 相当がない以上の実際的な対処。

## 代替案

- 例外: 使えなくはないが、「失敗を無視したい」場合の制御が面倒。
- std::expected のモナディック操作: 理論上は美しいが C++ の構文では実用的でない。
