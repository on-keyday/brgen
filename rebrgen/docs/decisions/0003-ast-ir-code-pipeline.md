# AST→IR→Code パイプラインの採用（EBM の導入）

## 日付

- 判断時期: 2024年末 〜 2026年初頭（BM→EBM への移行を含む長期的な変遷）
- 文書化: 2026-04-15

## 判断

brgen の元々の AST→Code モデル（json2xxx 系）から、
AST→IR(EBM)→Code モデル（ebmgen → ebm2xxx 系）に移行する。

## 動機

- **write once generate any language code**: json2xxx 系では言語に依存しない共通の展開処理が
  各ジェネレーター内で繰り返し実装されていた。結果としてあるジェネレーターではサポートしているが
  別のではしていない、という状況が発生していた。IR を挟むことで共通の展開を一箇所にまとめ、
  各言語ジェネレーターはコード出力に集中できるようになった。
- サポート状況がまばらな点は現在の ebm2xxx でも残っているが、基本的な展開は共通化されているため
  新言語の立ち上げ速度は上がっている。
- 統一的な IR があることで、言語間の比較や品質基準の統一がしやすくなった。

## 具体例

### json2xxx 時代の問題

- AST のトラバース処理が各 json2xxx で個別に実装されており、AST の変更に弱かった。
- 言語非依存の展開（制御フロー、エンディアン処理など）が各所に散在していた。

### BM (Binary Module) での学び

- BM の良かった点: 物理レイアウトがフラットだったため transform pass 的な処理がやりやすかった。
- BM の問題点: 論理レイアウトまでフラットにしていたため拡張性が低かった。
  ID が付いている node と付いていない node が混在していた。
- 最大の教訓: **普通のコンパイラのように意味論を削っていくと、後から復元するのが非常に困難**。
  機械語へのコンパイラなら IR で意味論を落としていくのが正しいが、
  brgen はトランスパイラであり出力先も高級言語。構造や意味を保ったまま渡す必要がある。
  コンパイラとトランスパイラでは IR の設計方針が根本的に異なる。

### EBM の設計

- 全 node が ID を持つ。
- 物理レイアウトは BM と同じくフラットに保ちつつ、ID 参照によるグラフ構造で
  AST をたどるという論理的な意味を素直に表現できるようにした。
- WeakRef は AST の `ast::Ident` の base のような参照関係をそのまま反映している。

### ebmcodegen

- BM 時代の gen_template が前身。各言語ごとのボイラープレート（トラバース処理等）を
  自動生成することで、開発者はフック実装に集中できる。

## これは X を意味しない

- ID 参照ベースなのは深い設計思想というより、extended_binary_module.bgn で定義する上での都合、
  ポインタ直挿しへの置き換え手法を当時知らなかったこと、json2cpp2 の生成の限界など
  実際的な制約の積み重ねによるもの。

## 代替案

- AST→Code を維持して共通処理をライブラリ化: 各言語の AST ライブラリ（ast2go, ast2ts 等）は
  既にあったが、トラバース処理の共通化には限界があった。
