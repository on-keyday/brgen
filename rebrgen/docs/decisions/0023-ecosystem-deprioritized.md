# エコシステム整備はコア安定後に本格化する

## 日付

2026-04-15

## 判断

現段階ではエコシステム整備（ドキュメントの充実、パッケージ管理への公開等）よりも、コード生成品質の向上（unictest 成功率の改善、テストケースの拡充）を優先する。エコシステム的要素として Web Playground と LSP は既に提供しているが、本格的な整備はコアが安定してから行う。

## 動機

- **write once generate any language code（多言語生成）**: コアバリューであるコード生成が十分な品質に達していない段階で、エコシステムを整備しても砂上の楼閣になる。まず各言語の生成コード品質を上げることが先決。
- **研究開発段階の現実**: brgen の定義言語仕様、EBM の IR 構造、ebmcodegen のフック体系いずれも変化が激しい。ドキュメントを書いても記述対象が unstable なため追従コストが高く、陳腐化が早い。
- **個人プロジェクトの制約**: 一人開発でリソースが限られる以上、優先順位をつけざるを得ない。

## 具体例

**既にやっていること:**

- Web Playground（https://on-keyday.github.io/brgen）を公開済み。ブラウザ上で .bgn を書いてコード生成を試せる
- LSP サーバーを提供済み（VS Code 拡張）
- 公開ドキュメント（https://on-keyday.github.io/brgen/doc/）を一応整備している
- unictest 結果を公開している（https://on-keyday.github.io/brgen/unictest-results/）
- ローカル API サーバー（`script/api_server.sh`）を提供済み。server-kit をダウンロードして `npm run serve` でコード生成 API をローカルで動かせる。MCP サーバーとしても機能する

**コア安定後に本格化すること:**

- AST ライブラリ（ast2go, ast2ts 等）のパッケージ管理システムへの公開 — 単体で公開してもほとんど意味がない。コード生成パイプライン全体があって初めて価値がある
- ドキュメントの網羅的整備 — 記述対象が安定するまで待つ。現状は最低限の導入ドキュメントのみ

**やる意思はあるがまだ手をつけていないこと:**

- CLI ツール（src2json, ebmgen, ebm2xxx 等）の apt/brew/scoop 等での配布 — ユーザーが使い始める入口として意味がある。コア安定を待つ必要は必ずしもなく、やり方を調べれば着手可能。GitHub Releases にはマルチプラットフォームのバイナリが既にあるが、rebrgen 系ツールは現状 Linux ビルドのみ。brew/scoop で配布するなら macOS/Windows ビルドの CI 追加が前提になる

## これは X を意味しない

- **エコシステムを軽視しているわけではない。** 将来的には必要だと認識している。順序の問題。
- **何もやっていないわけではない。** Web Playground、LSP、公開ドキュメント、unictest 結果公開は既にエコシステムの一部。
- **コアが「完成」するまで一切手をつけないわけではない。** ドキュメントも書けるところは書いている。ただし unstable な部分の網羅的整備は費用対効果が低い。

## 代替案

- **今すぐドキュメントとパッケージ公開を整備する**: 記述対象が安定していないため追従コストが高い。変更のたびにドキュメント更新が必要になり、コア開発の速度が落ちる。
- **外部コントリビューターを呼び込むためにエコシステムを先に整備する**: 魅力的だが、コアが不安定な状態でコントリビューターを集めても、破壊的変更で離反するリスクが高い。
