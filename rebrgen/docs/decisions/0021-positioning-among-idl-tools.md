# 既存 IDL/スキーマツールとの差分と brgen/rebrgen の立ち位置

## 日付

2026-04-15

## 判断

brgen/rebrgen は既存の IDL/スキーマツール群（Protocol Buffers, Kaitai Struct 等）と
同じ「スキーマ定義 → 多言語コード生成」の問題領域にあるが、
明示的な IR (EBM) を挟むアプローチを取っている点で異なる。
この差分を認識した上で設計判断を行う。

## 動機

- 問題領域自体は珍しくない。IDL → 多言語コード生成は確立されたパターン。
- しかし既存ツールの多くは AST→Code 直行型であり、
  明示的な IR を挟んでいるものは少ない。
- brgen 自身も旧 json2xxx 系では AST→Code 直行型だった。
  rebrgen で IR を導入した動機と、既存ツールとの構造的な違いを明確にしておく。

## 具体例

### 近隣のプロジェクトとの比較

**Kaitai Struct** — brgen に最も近い。バイナリフォーマット定義 (KSY/YAML) → 多言語パーサー生成。
- 構造: KSY → KS コンパイラ内部表現 → 各言語ジェネレーター（直行型）。
- brgen との差分:
  - Kaitai はデコード（読み取り）特化。エンコード（書き込み）は公式にはサポートしていない。
    brgen はエンコード/デコード両方を扱う。
  - Kaitai の KSY は YAML ベースの宣言的記述。brgen の .bgn はインデントベースの独自構文で、
    制御フロー（if/match/loop）や演算式を直接書ける。
  - Kaitai は IR を外部に公開しない。brgen/rebrgen は EBM を独立したバイナリ形式として
    外部に公開しており、ebmgen と ebm2xxx が分離している。

**Protocol Buffers / Thrift / Cap'n Proto** — IDL → 多言語コード生成の代表格。
- 構造: IDL → 各言語プラグインが直接生成（protobuf は descriptor という IR 的なものを持つ）。
- brgen との差分:
  - これらは RPC/シリアライゼーション用の IDL であり、既存のバイナリプロトコルの記述には使えない。
    「TCP ヘッダを定義する」「TLS レコードを定義する」といった用途には向かない。
  - brgen は既存のバイナリフォーマットの記述が主目的。自分でフォーマットを設計するのではなく、
    既にあるフォーマットを正確に記述してコードを生成する。
  - protobuf の descriptor は IR 的だが、コード生成ロジックは各言語プラグイン内に閉じており、
    共通の変換パスは持っていない。

**Zeek Spicy** — ネットワークプロトコルパーサー生成器。IR を挟むアプローチが brgen/rebrgen に最も近い。
- 構造: Spicy (.spicy) → HILTI IR → C++ コード → コンパイル → .hlto (共有オブジェクト)。
- brgen との共通点:
  - 明示的な IR (HILTI) を持ち、AST→IR→Code のパイプラインを採用している。
    この点で AST→Code 直行型の他ツールとは一線を画しており、アプローチとして最も近い。
  - バイナリプロトコルの解析が主目的である点も共通。
- brgen との差分:
  - 出力は C++ のみ。多言語生成はしない。Zeek への組み込みが主目的であるため、
    多言語対応の必要性がそもそもない。
  - HILTI/Spicy 両方にランタイムライブラリが必要。
    brgen は few runtime 方針 (ADR-0022) で生成コードの外部依存を最小化している。
  - Spicy の HILTI はコンパイラ的な IR（最終的に C++ に lowering される）であるのに対し、
    EBM はトランスパイラ的な IR（意味論を保ったまま各言語に渡す）。
    ADR-0003 の「コンパイラとトランスパイラでは IR の設計方針が異なる」がここに表れている。

**Haxe** — 1 つの汎用言語 → 複数ターゲット言語へのトランスパイル。
- 構造: Haxe AST → typed AST → 各ターゲットジェネレーター。
- brgen との差分:
  - Haxe は汎用プログラミング言語のトランスパイラ。brgen はバイナリフォーマット定義の DSL。
  - 問題の性質が根本的に異なるが、「意味論を保ったまま多言語に出す」という設計課題は共通。
  - Haxe の typed AST は brgen の EBM に近い役割を果たしている。

### brgen/rebrgen が明示的 IR を導入した理由

上記ツールの多くが AST→Code 直行型で十分に動いている中、
brgen/rebrgen が IR を導入したのは以下の事情による:

1. **共通処理の重複** (ADR-0003): json2xxx 系で言語非依存な処理が各ジェネレーターに散在していた。
   protobuf 等は各言語プラグインのメンテナーが別々にいるためこの問題が顕在化しにくいが、
   brgen は 1 人 + AI 体制なので重複が直接的な負担になった。
2. **transform pass の必要性** (ADR-0017): ビットフィールド解析や IO vectorization など、
   AST レベルでは扱いにくい変換がある。IR があることでこれらを言語非依存に行える。
3. **バイナリフォーマット定義の複雑さ**: protobuf 等の IDL は自分で設計したシンプルなフォーマットを
   扱うが、brgen は既存の複雑なプロトコル（TLS, DNS, ELF 等）を記述する。
   エンディアン処理、可変長フィールド、条件分岐を含むフォーマットの共通展開には
   IR レベルでの処理が有効だった。

## これは X を意味しない

- 既存ツールが AST→Code 直行型だから劣っているという主張ではない。
  protobuf や Kaitai Struct はそのアプローチで十分に成功している。
  brgen の事情（1 人開発、既存フォーマット記述、エンコード/デコード両対応）が
  IR の導入を動機づけた。
- brgen が既存ツールの上位互換だという主張ではない。
  protobuf は RPC エコシステム、Kaitai はフォーマットリバースエンジニアリングという
  それぞれの強みがあり、brgen とは用途が重なりつつも異なる。
- EBM のような明示的 IR が IDL ツール一般に必要だという主張ではない。
  問題の複雑さと開発体制によって最適解は異なる。

## 代替案

- 既存ツール（特に Kaitai Struct）を拡張して使う:
  Kaitai はデコード専用でありエンコードをサポートしていないため、
  brgen の目的には不十分だった。また .bgn の表現力（制御フロー、演算式）は
  KSY の宣言的記述では実現できない。
- AST→Code 直行型のまま共通処理をライブラリ化:
  json2xxx 時代に実質的にこのアプローチだったが、限界があった (ADR-0003)。
