# ジェネリクスは brgen middle pass で pre-monomorphize する

## 日付

2026-04-15

## 判断

ジェネリクスの具体化（monomorphize）を brgen の middle pass で行い、
EBM には具体化済みの型のみを流す。EBM/ebm2* はジェネリクスを知らなくてよい。

## 動機

- **enough to represent formats**: TLS の vector のように、長さプレフィックス付きデータで
  `len :u8; data :[len]u8` のようなパターンが data の型だけ異なる形で繰り返し出現する。
  ジェネリクスがないとこれらを個別に定義する必要があり、定義の重複が発生する。
- **write once generate any language code**: ジェネリクスのセマンティクスは言語間で大きく異なる
  （C++ テンプレート、Rust ジェネリクス、Go の型パラメータ、etc.）。
  各 ebm2* で個別にジェネリクス展開を実装するより、brgen 側で統一的に具体化したほうが
  各言語ジェネレーターの実装コストが大幅に下がる。

## 具体例

- `Foo<u8>` → `Foo_u8` としてマングル名で具体化済み型を生成
- raw ジェネリックテンプレートは ebmgen でスキップし、具体化されたクローンのみ EBM に出力
- sizeof 等の定数伝搬も具体型に基づいて解決済みの状態で EBM に入る

## これは X を意味しない

- 展開先の識別子名が不自然になる可能性がある（`Foo_u8` など）。
  各言語のイディオムに沿った名前にはならないが、正確さを優先している。
- 将来的にジェネリクスの情報（型パラメータ、制約など）を EBM に残し、
  ebm2* 側で言語ネイティブなジェネリクスとして出力する選択肢は排除していない。
  現時点では実装コストと複雑さのバランスから pre-monomorphize を選択している。

## 代替案

- EBM にジェネリクス情報を持たせて ebm2* 側で展開:
  各言語のジェネリクスに合わせた出力が可能になるが、
  全 ebm2* にジェネリクス展開ロジックが必要になり実装コストが高い。
  言語間のセマンティクス差異の吸収も各ジェネレーターの責務になる。
