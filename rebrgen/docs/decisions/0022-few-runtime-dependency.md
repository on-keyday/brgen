# 生成コードのランタイム依存を最小化する (few runtime)

## 日付

2026-04-15

## 判断

brgen/rebrgen が生成するコードは、ターゲット言語の標準ライブラリのみで動作することを目指す。
生成コードが専用のランタイムライブラリに依存しないようにする。

## 動機

- **導入障壁の低減**: 生成コードを使うためにランタイムライブラリのインストールや
  バージョン管理が必要になると、導入のハードルが上がる。
  標準ライブラリだけで動くコードなら、生成結果をそのままプロジェクトに組み込める。
- **ADR-0020 (few dependency core) と同じ思想の延長**: コア部分に外部依存を持ち込まないのと
  同様に、生成されるコード側でも依存を最小化する。
- **生成コードの透明性**: ランタイムの中で何が起きているかをユーザーが把握しやすい。
  ランタイムに隠蔽された処理がないので、生成コードを読めば動作が理解できる。

## 具体例

### Kaitai Struct との対比

Kaitai Struct は各言語にランタイムライブラリ（KaitaiStream 等）を要求する。
これは合理的なアプローチで、ストリーム操作やエンディアン変換などの共通処理を
ランタイムに集約することで生成コードをシンプルに保てる。
実際に参考になる設計判断は多い。

brgen は別のトレードオフを選んでいる:
- ランタイムが提供するような共通処理は、生成コード内にインライン展開するか、
  ターゲット言語の標準ライブラリの機能で直接表現する。
- 生成コードは長くなりうるが、外部依存なしで完結する。

### 実際の適用

- ebm2c: C 標準ライブラリすらほとんど使っていない。
  実際に eBPF 環境（標準ライブラリが満足に使えない）への移植実績がある。
  few runtime の方針が制約の厳しい環境での利用を可能にした具体例。
- ebm2python: Python 標準ライブラリのみ
- ebm2rust: std のみ
- ebm2go: 標準パッケージのみ

## これは X を意味しない

- すべてのターゲット言語ジェネレーターがこの方針に従うべきという主張ではない。
  あくまで自分（brgen/rebrgen 開発者）が作るジェネレーターの方針。
  サードパーティが EBM を使って独自のジェネレーターを作る場合、
  ランタイムライブラリを前提とするアプローチを取っても構わない。
- Kaitai Struct のランタイム方式が劣っているという主張ではない。
  ランタイムに共通処理を集約するのは賢明なアプローチであり、
  生成コードのサイズや可読性では有利な面がある。トレードオフの選択が異なるだけ。
- 生成コードが一切のヘルパー関数を含まないという意味ではない。
  生成コード内にヘルパー関数を含むことはありうるが、
  それは生成コードの一部であり外部ライブラリではない。

## 代替案

- ランタイムライブラリ方式: 共通処理をランタイムに集約し、生成コードはそれを呼ぶ。
  生成コードがシンプルになり保守しやすいが、ユーザーにランタイムの導入を要求する。
  Kaitai Struct がこのアプローチで成功しており、有効な選択肢ではある。
