## Project Overview

**brgen** (BinaRy encoder/decoder GENerator) is a binary format definition language and multi-target code generator. Users write `.bgn` format files describing binary protocols/file formats, which are parsed to a JSON AST by `src2json`, then fed to target-language code generators (`json2go`, `json2cpp2`, `json2rust`, `json2c`, `json2ts`, `json2kaitai`, `json2mermaid`, `json2vm`).

The pipeline: `.bgn` source ŌåÆ `src2json` (C++ parser) ŌåÆ JSON AST ŌåÆ `json2*` generators ŌåÆ target code.

The `tool/brgen` Go binary orchestrates this pipeline driven by `brgen.json`.

## Build Commands

### C++ (core parser + C++/C/TS/VM generators)

Requires: CMake Ōēź 3.26, Ninja, Clang (C++20). The `futils` dependency is auto-cloned into `utils/` by `script/clone_utils.sh`.

```bash
# Linux/macOS: configure + build (Debug by default)
python build.py native Debug

# Release
python build.py native Release

# WASM (Emscripten) build
python build.py web Release

# Build only (after configure)
ninja -C built/native/Debug install
```

Build output installs to `tool/` (native) or `web/dev/src/lib/` (web). CMake build files go to `built/<mode>/<type>/`.

### Go (json2go, brgen driver, json2kaitai, json2mermaid, gen_ast2*)

Module: `github.com/on-keyday/brgen`, Go 1.24.

```bash
go build -o tool/json2go ./src/tool/json2go
go build -o tool/brgen ./src/tool/brgen
go test ./...
go test ./src/tool/s2jgo/ -run TestName
```

### Rust (json2rust, unictest)

```bash
cargo build --manifest-path src/tool/json2rust/Cargo.toml
cargo test --manifest-path astlib/ast2rust/Cargo.toml
cargo test --manifest-path src/tool/json2rust/Cargo.toml
```

## Key Architecture

### Core pipeline (`src/core/`)
- `src/core/lexer/` ŌĆö tokenizer
- `src/core/ast/` ŌĆö AST node types (`ast.h`), parser (`parse.h`/`parse.cpp`), traversal (`traverse.h`), and tools in `ast/tool/`
- `src/core/middle/` ŌĆö type analysis/resolution pass (runs after parsing)
- `src/core/common/` ŌĆö shared utilities

### Code generators (`src/tool/`)
- `src2json` ŌĆö C++ binary: parses `.bgn` ŌåÆ JSON AST. Main entry for the whole pipeline.
- `json2cpp2`, `json2c`, `json2ts`, `json2vm`, `json2vm2` ŌĆö C++-implemented generators
- `json2go`, `json2kaitai`, `json2mermaid`, `json2graphviz` ŌĆö Go-implemented generators
- `json2rust` ŌĆö Rust/wasm-pack generator
- `brgen` ŌĆö Go driver binary that reads `brgen.json` and dispatches to generators
- `gen/` ŌĆö tools that **generate** the AST manipulation libraries (`gen_ast2go`, `gen_ast2ts`, `gen_ast2rust`, etc.)

### Auto-generated files ŌĆö do not edit manually
- `astlib/ast2go/`, `astlib/ast2ts/`, `astlib/ast2rust/`, `astlib/ast2py/`, `astlib/ast2c/` ŌĆö AST libraries generated by `gen_ast2*` tools in `src/tool/gen/`
- `web/dev/src/lib/bmgen/bm_caller.js`, `ebm_caller.js`, `bm_workers.js`, `ebm_workers.js` ŌĆö BM/EBM worker registries
- Files with `// Code generated by gen_ast2...; DO NOT EDIT.` headers

### Web Playground (`web/dev/`)
Framework: Preact + Zustand + Monaco Editor + Vite + TypeScript.

Worker architecture (browser mode):
- `GeneratorService` (`generator_service.ts`) ŌĆö top-level orchestrator; initialized once, shared across app
- `WorkerFactory` (`s2j/worker_factory.ts`) ŌĆö registry of `WorkerType ŌåÆ () => IWorker`
- `fixedWorkerMap` (`s2j/workers.ts`) ŌĆö core workers (src2json, json2cpp2, json2go, etc.), each as `() => new Worker(...)`
- `bm_workers.js` / `ebm_workers.js` ŌĆö BM/EBM workers from rebrgen, loaded dynamically
- `JobManager` (`s2j/job_mgr.ts`) ŌĆö wraps an `IWorker`, manages async job dispatch via `postMessage`/`onmessage`
- `EmWorkContext` / `GoWorkContext` ŌĆö run Emscripten/Go WASM inside a Web Worker thread
- `RequestQueue` (`s2j/request_queue.ts`) ŌĆö sets `globalThis.onmessage` inside worker threads; only for Worker context

Worker types by runtime:
- **Emscripten WASM** (src2json, json2cpp2, json2c, json2ts): `ccall("emscripten_main", ...)` with URL-encoded args
- **Go WASM** (json2go, json2kaitai): `go.json2goGenerator(sourceCode, args)`
- **wasm-pack/Rust** (json2rust): `json2rust(sourceCode)` directly

Server mode (`web/dev/src/server/`):
- `DirectWorker` ŌĆö implements `IWorker` by calling handler functions directly (no Web Worker threads)
- `createEmscriptenHandler` / `createGoHandler` / `createRustWasmHandler` ŌĆö create handlers using Node.js `import()`
- `GeneratorService` accepts optional `customWorkerMaps` to swap in `DirectWorker`-based maps
- HTTP API served by Hono: `POST /api/generate`, `GET /api/languages`
- Run with `npm run serve` (uses `tsx` for execution, not Vite)

Vite config notes:
- `node:fs`, `node:path`, `node:module` are shimmed to empty for browser builds
- `ast2ts` is a CJS symlink (`astlib/ast2ts/out/`), needs special `include` in `optimizeDeps` and `commonjsOptions`
- WASM workers need COOP/COEP headers (`Cross-Origin-Opener-Policy: same-origin`)
- Build output goes to `web/public/` (not `web/dev/dist/`)

### BM/EBM integration (rebrgen)

`bmgen` and `ebmgen` tools come from https://github.com/on-keyday/rebrgen and are integrated via CI. The files in `web/dev/src/lib/bmgen/` and `rebrgen/` are for local development. These auto-generated JS/WASM files must not be committed manually ŌĆö they are produced by the CI pipeline.

### LSP (`lsp/`)
- VS Code extension with a language server
- Client: `lsp/client/`, Server: `lsp/server/`
- Type analysis implementation lives in `astlib/ast2ts/`

### `brgen.json` schema

```json
{
  "src2json": "<path to src2json binary>",
  "libs2j": "<path to libs2j>",
  "suffix": ".bgn",
  "input_dir": ["./example/"],
  "warnings": { "disable_untyped": true, "disable_unused": true },
  "output": [
    { "generator": "./tool/json2cpp2", "output_dir": "./ignore/example/cpp2/", "args": [...] }
  ],
  "test_info_output": "./ignore/example/test_info.json"
}
```

## Code Style

### C++ (C++20, Clang)
- Every file starts with `/*license*/`
- `#pragma once` for headers
- `PascalCase` for types; `snake_case` for methods/variables/fields
- No exceptions; use `futils::error::Error<>` return types
- 4-space indent, same-line opening brace
- Namespace `brgen::`, sub-namespaces `brgen::ast`, `brgen::lexer`

### Go
- Standard `flag` package (not cobra/pflag)
- `// TODO(on-keyday):` for TODO comments
- Named imports when colliding: `ast2go "github.com/on-keyday/brgen/astlib/ast2go/ast"`

### Rust
- `anyhow::Result` with `?` propagation
- `clap` with derive macros for CLI
- `Rc<RefCell<T>>` aliased as `SharedPtr<T>` in AST code

### TypeScript
- ES2022, DOM lib, bundler moduleResolution
- `camelCase` for variables/functions, `PascalCase` for classes/interfaces

## Branch Naming Convention

- `lang/` ŌĆö language spec / parser changes
- `gen/` ŌĆö code generator changes
- `ast/` ŌĆö AST library / code generator for AST libs
- `web/` ŌĆö Web Playground changes
- `lsp/` ŌĆö LSP server changes
- `env/` ŌĆö CI, Dockerfile, scripts
- `tool/` ŌĆö other tooling
- `doc/` ŌĆö documentation
- `sample/` ŌĆö `.bgn` format examples

## Important Notes

- `ignore/` is a temporary output area (git-ignored); `built/` holds CMake build artifacts
- `spec/` is a remnant of specification work
- `testkit/` and `src/tool/cmptest` are deprecated ŌĆö use `src/tool/unictest` instead (currently used in rebrgen, not actively in brgen itself)
- `script/` contains miscellaneous helper scripts
- `example/` has 100+ real `.bgn` format definitions (UDP, TCP, DNS, TLS, ZIP, ELF, etc.)
- External dependency `futils` (github.com/on-keyday/utils) is cloned into `utils/` by build scripts
